Database requirements:

- A citizen has personal information (`CitizenProfile`) as follows: full name (`fullName`), date of birth (`dateOfBirth`), gender (`gender`), address (`address`), email (`email`), phone number (`phone`), national ID number (`cccdNumber`), avatar (`avatarUrl`).
- A citizen has one medical record (`MedicalRecord`) including: distinguishing marks (`distinguishingMarks`), blood group (`bloodGroup`), allergies (`allergies`), background diseases (`backgroundDiseases`), current medications (`currentMedications`), notes (`notes`).
- A citizen may have multiple emergency contacts; each emergency contact includes: full name, address, phone number 1, phone number 2, and relationship to the citizen.
- A citizen can manage NFC tags. Each NFC tag has: NFC tag id (the tag's serial code), name, tag type, status, registration date, last used time (`last_used_at`), and the tag owner (stored as `citizen_id`). If a citizen loses an NFC tag, they can report it lost and request a new one. A citizen can also request to cancel an old tag.
- A citizen can manage QR codes. Each QR code has: QR code id, name, status, creation date, last used time (`last_used_at`), and the creator (stored as `citizen_id`). If a citizen loses a QR code, they can disable the old one and create a new QR code. A citizen can also request to cancel an old QR code.
- The information stored on a QR code or NFC tag is the result of a hash function over the citizen's id and a system secret. Specifically, for QR codes, the stored value is a JSON containing the QR code's id and the system-hashed citizen id. For NFC tags, the stored value is the system-hashed citizen id.
- NFC/QR identification flow: scan NFC/QR → the system looks up the NFC/QR row by the tag/QR code id → check the NFC/QR status → 1. if the status is inactive, the system returns nothing; 2. if the status is active, the system takes `citizen_id` from the NFC/QR row and returns the citizen's information → then compares against the citizen's hash_id to confirm it is indeed that citizen. If it does not match, return nothing. If it matches, return the citizen's information.
- The citizen Hash_id must NEVER be stored anywhere in the system, in any table.
